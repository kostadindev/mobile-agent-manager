AI Agents are increasingly capable of performing personal and professional tasks~\cite{Kim2025}, which is motivating people to assemble personalized teams of agents. However, currently these agents need to be controlled via a programming interface or specialized desktop applications. However, the main value of these agents is that they can serve us semi- or fully autonomously while we are away from the workplace or a desktop computer. This has created strong demand for mobile applications that allow people to control their own agents while away from their workstations. This was demonstrated by the strong response to ClawdBot~\cite{Clawdbot2025}, an open-source autonomous AI agent controlled via messaging platforms such as WhatsApp. Yet Clawdbot operates as a single agent with difficult-to-control permissions, planning, and approval. Additionally, existing agentic management platforms, such as OpenAI Frontier and blah blah, are both commercial and not optimized for mobile use while on the go. This underscores the need for a fully open-source, fully mobile agent management platform that enables people to manage their custom agentic workflows with adjustable levels of control and governance while on the go.

We designed a mobile-first platform that lets users \emph{bring their own AI agents} and control them on the go through text, voice, and images (Figure~\ref{fig:flow}). The architecture is \textbf{agent-agnostic}. An LLM orchestrator breaks down requests, assigns tasks to agents, and waits for them to complete before returning a cohesive result. Multimodality is key in this design. Text is sent via a standard conversational interface that supports asynchronous communication with agents. Voice enables hands-free control while walking or cycling. Images allow you to snap a photo in the environment and send it as context to the agents. Given that these workflows may execute tasks with varying impact and risks, we prioritize having a highly customizable level of control. Certain tasks will be executed completely autonomously, while others may require approval, additional input, or real-time monitoring. [INSERT HOW THIS APP Is evaluated]. Multimodality and control are essential not only for providing the necessary flexibility to assemble custom multi-agent systems but also to enable mobility and easy-to-use interaction.

We envision that mobile control of custom multi-agent systems will be essential in modern human-AI interaction and key to effective management of personal and professional agents. This open-source application is initially targeted at people with a technical background who already use agents in their day-to-day life. Examples may include academics, managing agents for explorative research, summarization, and documentation, software engineers remotely steering teams of long-running coding agents, or quantitative analysts who use agents for trading and financial data analysis. As agents become increasingly capable in more domains and familiar to non-technical users, the design of this application can be expanded to target anyone who utilizes AI.

\subsection{Design Rationale}

To begin, I drew on human-centred AI principles from the lectures (reliability, safety, transparency)~\cite{Shneiderman2020} and looked online for interaction guidelines that emphasise: (a) setting expectations, (b) enabling efficient correction/repair, and (c) communicating uncertainty~\cite{Amershi2019}. I also used Value Sensitive Design (VSD)~\cite{Friedman2002} as inspiration into how to translate stakeholder values into concrete interaction requirements.

\textbf{V1: Fast} ``It's a simple task; it should just do it instead of asking for permission.''

\textbf{V2: Control} ``There should be guardrails and a control mechanism for sensitive tasks.''

\textbf{V3: Mobility:} ``I should be able to reach my agents hands-free.''

\textbf{V4:}

From these values, I drafted some initial resulting design implications, using methods from VSD~\cite{Friedman2002} and related work~\cite{vanDePoel2013}:

\textbf{D1:} Design a setting giving the agent full autonomy for low-r\textbf{D2:} Propose a plan and ask for confirmation before executing in plan mode; Propose plan and render a real-time execution graph showing active processes (Figure~\ref{fig:configs}).

\textbf{D3:} Support customization of orchestration policy and output stylization (Appendix~\ref{app:backstories}).

\textbf{D4:} Provide the ability to turn agents on and off in the interface (Figure~\ref{fig:ss-agents}).

\textbf{D5:} Support hand-free voice-only mode (Figure~\ref{fig:chat}c).

\textbf{D6:} Support images for quick snaps of environmental context (whiteboards, notes, etc) (Figure~\ref{fig:chat}b).

I used these to begin thinking about the prototype, which I called `Mobile Agents'. We support D1 and D2 via a control setting that includes Blackbox, Plan, and Transparency modes (Figure~\ref{fig:configs}; Table~\ref{tab:transparency}). In. Blackbox gives the system full autonomy to execute tasks. Planning mode constructs a structured plan and asks the user for permission before executing. Transparency mode does everything Plan mode does and provides a real-time graph of execution traces for full visibility into the task. Additional control is implemented through D3 and D4. D3 is implemented through a natural-language constitution provito the Orchestrator agent in the system prompt to guide orchestration behavior and output style. D5 and D6 are exercised through the modality setting. We support text and images, native voice input/output (push-to-talk), and multimodal (speech-to-text, text, and images) (Figure~\ref{fig:chat}). The text input is the baseline interaction with the multi-agent system and supports all control settings, mobile, and visualizations. Images are added to the text and are useful for capturing physical context (e.g., whiteboard, notes) using the device's camera. The voice-only setting enables hands-free use but does not support plan and transparency controls. Multimodal is similar to text and images mode but also supports speech input that is converted to text, which does not limit plan and transparency modes.

\subsection{Accessibility and Inclusion}

HCAI demands inclusivity across age, language, culture, and ability levels~\cite{Capel2023}. MobileAgents supports English, Chinese, Arabic with full RTL layout, and Bulgarian. It has dark, light, and auto color themes (Figure~\ref{fig:ss-settings}). Icon buttons contain ARIA labels. Users with different abilities and devices can utilize the product through multimodal inputs of text, voice, and images, with neither being a point-of-failure. The design was chosen with personalization in mind and to not impose a single interaction pattern.

\section{Methods: Research Question, Evaluation Methods, and Metrics}
%% ============================================================

\subsection{Research Question}

\textbf{RQ1:} Is there an efficiency/control tradeoff between the Blackbox, Plan, and Transparency modes?

\textbf{RQ2:} Does the support of multiple modalities (text, voice, images) improve the mobile-first experience?

\textbf{RQ3:} Are users able to tune the orchestration behavior through the constitution and on/off switches in the agent management tab (Figure~\ref{fig:ss-agents})?

\subsection{Study Design}

The study runs in three sequential phases, each targeting one RQ. All phases use a within-subjects design; order effects are controlled by counterbalancing with a Latin square.

\textbf{Phase 1 --- Transparency modes (RQ1).} Each participant completes the same research task (find and compare two recent papers on an assigned topic) under all three transparency configurations (Table~\ref{tab:transparency})---Blackbox (C1), Plan Preview (C2), and Full Transparency (C3)---in counterbalanced order. The task is held constant so that any difference in time, approval behaviour, or subjective ratings is attributable to the configuration, not the task.

\textbf{Phase 2 --- Modality comparison (RQ2).} Each participant completes a fixed multi-step workflow (find related work and draft a short proposal) using each of the three input modalities in counterbalanced order: (M1)~text only, (M2)~image (photograph a printed abstract), (M3)~voice dictation (Figure~\ref{fig:chat}). All three runs use Plan Preview (C2) as the transparency baseline so that modality is the only variable. This controls for configuration effects identified in Phase 1.

\textbf{Phase 3 --- Agentic customisation (RQ3).} Participants receive an open-ended scenario: ``Configure the system so that it never sends Slack messages without showing you a draft first, and remove any agents you would not want running automatically.'' They are then asked to run a multi-step query and observe whether the system respects their settings. This phase is not counterbalanced; it is a single exploratory session designed to surface whether participants can locate, understand, and apply the constitution editor and agent on/off toggles (Figure~\ref{fig:ss-agents}).

\textbf{Tasks.}
\begin{itemize}
  \item \textbf{T1} (Phase 1 \& 2 baseline): Type a text query to find and compare two recent papers on a given topic. Exercises ArXiv and Semantic Scholar agents.
  \item \textbf{T2} (Phase 2, M2): Photograph a printed paper abstract with the device camera and ask the system to find related work and draft a proposal. Exercises image analysis, ArXiv, Wikipedia, and Proposal agents.
  \item \textbf{T3} (Phase 2, M3): Use voice input to dictate a request to summarise results and send them to a Slack channel. Exercises voice transcription and the Slack agent with checkpoint approval.
  \item \textbf{T4} (Phase 3): Edit the orchestrator constitution and toggle at least one agent off, then rerun T1 and verify the change in behaviour.
\end{itemize}

\subsection{Participants}

We recruited 13 Cambridge students. All had experience with LLM-based tools. Participation was voluntary.

\subsection{Evaluation Methods}


In phase 1, participants completed a 7-point Likert survey for  \emph{perceived control} and \emph{satisfaction} after each control setting. In phase 2, they also completed the NASA Task Load Index ~\cite{Hart1988} to test for cognitive load across modalities, and the Single Ease Question~\cite{Sauro2009} for usability. Participants rated the constitution and agent toggles on a 3-item perceived controllability scale. They shared live comments on the user experience as they tried out the application. An interview (questions in Appendix~\ref{app:interview})) explores transparency and modality preferences across RQ1 and RQ2. Additionally, it queries the usability and usefulness of agent controls for RQ3.



\subsection{Metrics}

\textbf{RQ1 --- Efficiency/control tradeoff:}
\begin{itemize}
  \item Trust, perceived control, and satisfaction (Likert, 1--7).
  \item Qualitative opinions on over- or under-informed cases.
\end{itemize}

\textbf{RQ2 --- Multimodal mobile experience:}
\begin{itemize}
  \item NASA-TLX composite score and subscale scores per modality.
  \item Single Ease Question (SEQ).
  \item Modality preference between different contexts.
\end{itemize}

\textbf{RQ3 --- Orchestration tuning:}
\begin{itemize}
  \item Task success after customisation: binary measure of whether the system respected the user's constitution rule and agent configuration during T4.
  \item Perceived controllability scale score (3-item Likert).
  \item Qualitatively on discoverability, understanding, and ownership.
\end{itemize}

\subsection{Qualitative Assessment}

Analysis of think-aloud and interviews was grouped into the following:

\textbf{Intervention is better than observation.}
Participants valued the ability to \emph{act}---approving, rejecting, or modifying a plan---more than passively watching execution. One participant noted: ``Seeing the graph is cool, but what actually makes me trust it is that I can stop it before it sends the Slack message.'' This directly validates Amershi et al.'s~\cite{Amershi2019} G2 (``support efficient correction'') and G18 (``support efficient dismissal''). The quantitative data reflects this: control scores rose sharply from C1 (3.2) to C2 (5.4) (Table~\ref{tab:results}) even though C2 adds only a plan card, not a live graph---suggesting that the \emph{ability to intervene} matters more than execution visibility.

\textbf{High visibility is valued}

While the participants expressed enthusiasm about AI capabilities, the trust in their behavior is still low. Thus, the high level of observability in transparency mode is valued, especially in tasks that involve irreversibile actions like sending out Slack messsages. This is also supported by the quantative assessment in RQ1 (Table~\ref{tab:results}; Table~\ref{tab:posthoc-rq1}).



\textbf{Theme 3: Multimodal enables mobile}
Participants strongly valued voice for on-the-go use. but consistently avoided it in shared or public spaces. However, they were hesitant to use it in public spaces and for sensitive tasks. Furthermore, images enable capture of the environment that would not be possible to easily and quickly describe in text. Text remains useful as the backup modality for more complex tasks.

\textbf{Theme 4: Constitution is confusing; agent toggles are not.}
Users struggled with using the constitution feature because it is difficult to discover and not immediately clear how to use. However, they found the agent toggles simple, easy, and effective to build teams.

\section{Discussion & Future Work}
%% ============================================================

\subsection{Ethical, Social, and Value-Related Reflection}



\textbf{Automation vs.\ autonomy on mobile.}

Our findings surface the difficulty of designing an agentic application that prioritizes both efficiency and control. Users want agents to be autonomous but do not want to give up control. Black Box scores lowest on trust and control. Yet, certain users preferred it for low-stakes queries. This issue is especially critical in mobile settings, as each additional tap undermines the motivation for ambient, on-the-go assistance. The transparency mode received the most trust. However, we recognize that the live graph may create an illusion of control and oversight. Overall, we chose to support all three settings and make switching easy, with transparency mode as the default. Future designs can explore an auto mode that can classify individual prompts into the appropriate control setting.

\textbf{Privacy & Security}

We recognize that a single centralized agentic manager brings risks for data storage and prompt injection from malicious agents. Thus, we release the tool as an open-source solution so that individual users and organizations can self-host on-premises and own all of their data. The current system is vulnerable to adversarial attacks from the integrated agents. In the current version, we recommend bringing only trusted agents. Future work should research the risks and integrate guardrails and safety mechanisms to prevent unintended behavior.

\textbf{Inclusivity and access.}

Mobile Agents is built around inclusivity and broad access through deep customization. We support multiple modalities, multiple languages, RTL layout, and light/dark themes. However, our design and testing have been limited in scope and size and is not currently representative of the wide range of potential users. Users with lower technical literacy may struggle to utilize our architecture. Certain features like the constitution and bring-your-own agent archiecture is currently only accessible to power users. While we have a tab with documentation, it should be expanded to cover the topics in simpler, more accessible language. We should support easier integration with agents through MCP servers and expand current accessibility coverage.


\textbf{Conclusion.} MobileAgents demonstrates that personal AI agents can be made controllable from a mobile device through multimodal input (text, voice, camera) and an agent-agnostic orchestration layer. The bring-your-own-agents architecture means users are not locked into a fixed agent set---they plug in whatever tools matter to them and manage them from their phone. Configurable transparency lets users match oversight to context: Plan Preview for quick mobile use, Full Transparency for high-stakes actions. The ability to \emph{intervene}---not merely observe---drove trust more than visual complexity. Consent-gated multimodality respected privacy in mobile contexts, though speech recognition equity and automation bias remain open challenges. These findings contribute to the design space of mobile, multimodal, user-controllable multi-agent AI systems.

\section{Authentication and Data Persistence}
\label{app:auth}

\subsection{Authentication}

The server uses Supabase for authentication.

\begin{enumerate}
  \item Extracts the Bearer token from the \texttt{Authorization} header.
  \item Validates it via \texttt{supabase.auth.get\_user(token)}, which works with both HS256 and ES256 JWT keys.
  \item Returns \texttt{\{sub: user\_id, email: user\_email\}} or raises HTTP~401.
\end{enumerate}

The client stores the Supabase session token and includes it in all API requests.

\subsection{Server-Side Persistence}

Conversations, messages, execution records, and approval decisions are persisted to Supabase (PostgreSQL). The database layer (\texttt{services/db.py}) provides:

\begin{itemize}
  \item \textbf{Conversations:} Create, list (by user), get, delete. Each conversation stores a title and transparency level.
  \item \textbf{Messages:} Insert and list (by conversation). Each message stores role, content, optional image URL, voice transcript, agent ID, input modality, task plan, and execution graph.
  \item \textbf{Executions:} Create (with plan and graph snapshots) and complete (with status, summary, and step results).
  \item \textbf{Approvals:} Create, resolve (approve/reject with optional comment), and query by step ID.
\end{itemize}

\section{Survey Instrument}
\label{app:survey}

All Likert items use a 7-point scale (1 = Strongly Disagree, 7 = Strongly Agree) unless otherwise noted.

%% --- S0 ---
\subsection*{Part 0: Background (before study begins)}

\begin{enumerate}
  \item Age range. \emph{(18--21 / 22--25 / 26--30 / 31--40 / 40+)}
  \item Gender. \emph{(Male / Female / Non-binary / Prefer not to say)}
  \item How often do you use AI assistants (e.g., ChatGPT, Copilot)? \emph{(Daily / Several times a week / Weekly / Monthly / Rarely or Never)}
  \item Have you used a multi-agent AI system before? \emph{(Yes / No / I'm not sure)}
  \item Which AI tools do you use regularly? \emph{(Select all: ChatGPT, Copilot, Gemini, Claude, Cursor, Perplexity, Other)}
\end{enumerate}

%% --- S1 ---
\subsection*{Part 1: After each transparency condition (C1, C2, C3) --- RQ1}

Completed three times, once per configuration. Items 1--3 are primary metrics; items 4--7 are supplementary.

\begin{enumerate}
  \item I trusted the system's output enough to act on it. \emph{(Trust)}
  \item I felt in control of what the system did on my behalf. \emph{(Perceived Control)}
  \item I am satisfied with this interaction. \emph{(Satisfaction)}
  \item The system made it clear what it was going to do before doing it. \emph{(Expectation Setting)}
  \item It was easy to correct or stop the system when needed. \emph{(Repairability)}
  \item I understood why the system made the decisions it did. \emph{(Explainability)}
  \item I felt comfortable with the information I shared during this task. \emph{(Privacy Comfort)}
  \item[--] Any comments about this mode? \emph{(open text, optional)}
\end{enumerate}

After all three conditions:
\begin{enumerate}
  \item[8.] Which transparency mode would you prefer for everyday use? \emph{(C1 Blackbox / C2 Plan Preview / C3 Full Transparency)}
  \item[9.] Which mode would you prefer for high-stakes tasks (e.g., sending a Slack message on your behalf)? \emph{(C1 / C2 / C3)}
  \item[10.] Why did you prefer that mode? \emph{(open text, optional)}
\end{enumerate}

%% --- S2 ---
\subsection*{Part 2: After each modality condition (M1, M2, M3) --- RQ2}

Completed three times, once per modality (text, image, voice). The NASA-TLX uses a 1--10 scale (Very Low--Very High); Performance is reversed (1 = Perfect, 10 = Failure).

\textbf{Single Ease Question (SEQ):}
\begin{enumerate}
  \item Overall, how would you rate the difficulty of this task? \emph{(1 = Very Difficult, 7 = Very Easy)}
\end{enumerate}

\textbf{NASA Task Load Index~\cite{Hart1988}:}
\begin{enumerate}
  \item[2.] Mental Demand --- How much mental and perceptual activity was required? \emph{(1--10)}
  \item[3.] Physical Demand --- How much physical activity was required? \emph{(1--10)}
  \item[4.] Temporal Demand --- How much time pressure did you feel? \emph{(1--10)}
  \item[5.] Performance --- How successful were you in accomplishing the task? \emph{(1 = Perfect, 10 = Failure)}
  \item[6.] Effort --- How hard did you have to work to accomplish the task? \emph{(1--10)}
  \item[7.] Frustration --- How irritated, stressed, or annoyed did you feel? \emph{(1--10)}
\end{enumerate}

\textbf{Modality comfort (7-point Likert):}
\begin{enumerate}
  \item[8.] Using this modality felt natural for this type of task. \emph{(Modality Comfort)}
  \item[9.] I would use this modality in a public or shared space. \emph{(Context Appropriateness)}
  \item[--] In what situations would you choose or avoid this modality? \emph{(open text, optional)}
\end{enumerate}

After all three modality conditions:
\begin{enumerate}
  \item[10.] Which input modality would you use most often on a mobile device? \emph{(Text / Voice / Image / It depends)}
  \item[11.] Which modalities felt well-suited to on-the-go use? \emph{(Select all: Text / Voice / Image)}
  \item[--] Was there a modality that surprised you -- positively or negatively? \emph{(open text, optional)}
\end{enumerate}

%% --- S3 ---
\subsection*{Part 3: After the customisation task (Phase 3) --- RQ3}

\begin{enumerate}
  \item I was able to find the constitution editor without help. \emph{(Discoverability)}
  \item I understood what the constitution field does. \emph{(Mental Model)}
  \item Editing the constitution changed the system's behaviour as I expected. \emph{(Effectiveness)}
  \item The agent on/off toggles were easy to understand and use. \emph{(Controllability)}
  \item After customising the system, I felt more in control. \emph{(Perceived Ownership)}
  \item I would use the constitution and agent controls regularly. \emph{(Adoption Intent)}
  \item[7.] Which type of change did you make to the constitution? \emph{(Constraint / Style / Routing / Did not edit / Other)}
  \item[--] Describe what you changed and whether the system behaved as expected. \emph{(open text)}
\end{enumerate}

%% --- S4 ---
\subsection*{Part 4: Overall post-study feedback}

\begin{enumerate}
  \item Overall, I would trust this system to act on my behalf on a mobile device. \emph{(Overall Trust)}
  \item I felt this system respected my privacy. \emph{(Privacy)}
  \item I would use this application in my day-to-day work. \emph{(Adoption Intent)}
  \item[4.] Which single feature contributed most to your trust? \emph{(Plan Preview / Checkpoint approval / Live DAG / Agent on/off / Constitution / Multimodal input)}
  \item[5.] Which single feature caused the most friction or concern? \emph{(Plan approval steps / Live graph overwhelming / Voice risky in shared spaces / Camera awkward / Agent controls hard to find / Constitution unclear)}
  \item[--] What is the single most important improvement you would make? \emph{(open text)}
  \item[--] Any other comments or suggestions? \emph{(open text)}
\end{enumerate}
